# ADR-990：文档演进路线图管理规范

> ⚖️ **本 ADR 是文档体系演进路线图的唯一管理标准，定义路线图的结构、更新机制和与 ADR 的关联。**

**状态**：✅ Accepted  
**版本**：1.0
**级别**：文档演进管理 / 治理层  
**适用范围**：所有文档体系演进计划  
**生效时间**：即刻

---

## Focus（聚焦内容）

- 路线图结构与组织
- 季度更新机制
- 与 ADR/RFC 的关联
- 状态追踪
- 透明度与可访问性

**下次更新**：YYYY-MM-DD（每季度末）

**组织原则**：
- **当前季度**：详细进度，包含百分比和负责人
- **下一季度**：计划项目，包含负责人
- **未来季度**：方向性规划
- **长期愿景**：战略目标，不具体到项目

**核心原则**：
> 清晰分层，近期详细，远期方向性。

**判定**：
- ❌ 路线图内容混乱，无层级
- ❌ 缺少负责人或状态
- ✅ 结构化组织，清晰可追踪

---

### 季度更新机制（ADR-990.2）

**规则**：

路线图 **必须**每季度更新一次，更新时间为季度末最后一周。

**更新流程**：
```markdown
1. 提前准备（季度末前 2 周）
   - 收集团队输入
   - 评估当前季度完成情况
   - 识别下季度优先级

2. 提案（季度末前 1 周）
   - 创建 RFC：`RFC-ROADMAP-YYYY-QX.md`
   - 提议新增/调整项目
   - 团队讨论和投票

3. 架构评审（季度末）
   - 架构委员会审查
   - 批准下季度计划
   - 确定负责人和里程碑

4. 更新路线图
   - 更新 `docs/ROADMAP.md`
   - 移动完成项目到"已归档"
   - 推进"下一季度"到"当前季度"
   - 添加新的"未来季度"项目

5. 通知团队
   - 发布更新公告
   - 通知所有负责人
   - 更新相关 Issue/PR
```

**更新清单**：
```markdown
## 季度路线图更新清单

季度：YYYY QX
更新人：[姓名]
更新日期：YYYY-MM-DD

### 当前季度回顾
- [ ] 识别所有已完成项目
- [ ] 评估未完成项目（原因和下一步）
- [ ] 更新完成百分比
- [ ] 移动完成项目到"已归档"

### 下一季度规划
- [ ] 确定优先级（P0/P1/P2）
- [ ] 分配负责人
- [ ] 估算工作量
- [ ] 检查依赖关系

### 文档更新
- [ ] 更新 `docs/ROADMAP.md`
- [ ] 更新相关 ADR 的状态
- [ ] 更新项目 README（如适用）

### 通知
- [ ] 创建公告 Issue
- [ ] 通知所有相关负责人
- [ ] 更新团队日历
```

**核心原则**：
> 固定节奏，团队参与，透明更新。

**判定**：
- ❌ 路线图长时间未更新
- ❌ 更新流程不透明
- ✅ 每季度按流程更新

---

### 与 ADR/RFC 关联（ADR-990.3）

**规则**：

路线图项目 **必须**标注对应的 ADR 或 RFC 编号。

**关联格式**：
```markdown
### 进行中（In Progress）
- [ ] [ADR-940](../adr/governance/ADR-940-adr-relationship-traceability-management.md): ADR 关系与溯源管理 - @owner - 80%
- [ ] [RFC-001](../rfcs/RFC-001-doc-search-optimization.md): 文档搜索优化方案 - @owner - 30%
```

**反向关联**：

ADR/RFC **应该**在元数据中标注其在路线图中的位置：

```markdown
# ADR-940：ADR 关系与溯源管理宪法

**状态**：✅ Accepted
**版本**：1.0
**路线图**：2026 Q1 - 进行中（80%）
```

RFC 示例：
```markdown
# RFC-001：文档搜索优化方案

**状态**：📋 Draft
**版本**：1.0
**路线图**：2026 Q2 - 计划中
```

**状态同步**：
- ADR 状态变更时，更新路线图中的状态
- 路线图项目完成时，更新 ADR 元数据

**核心原则**：
> 路线图与 ADR/RFC 双向关联，状态同步。

**判定**：
- ❌ 路线图项目无对应 ADR/RFC
- ❌ ADR 完成但路线图未更新
- ✅ 双向关联，状态同步

---

### 状态追踪（ADR-990.4）

**规则**：

路线图项目 **必须**定期更新进度百分比和状态。

**状态类型**：
- **进行中（In Progress）**：已启动，正在开发
- **计划中（Planned）**：已批准，等待启动
- **已完成（Completed）**：已交付并合并
- **已暂停（On Hold）**：暂时停止，等待条件
- **已取消（Cancelled）**：不再进行
- **已废弃（Deprecated）**：被替代或失效

**进度更新频率**：
- **P0 项目**：每周更新
- **P1 项目**：每两周更新
- **P2 项目**：每月更新

**进度计算**：
```markdown
百分比 = (已完成工作 / 总估算工作) × 100%

示例：
- [ ] ADR-940: ADR 关系与溯源管理 - @owner - 80%
  - [x] 定义关系类型
  - [x] 编写规则正文
  - [x] 创建架构测试
  - [ ] 创建自动化工具
  - [ ] 更新现有 ADR
```

**里程碑标记**：
```markdown
- [ ] ADR-980: 生命周期同步机制 - @owner - 60% - 🎯 Milestone: 2026-02-15
```

**核心原则**：
> 实时进度，透明追踪，明确里程碑。

**判定**：
- ❌ 项目进度长时间未更新
- ❌ 进度百分比不准确
- ✅ 定期更新，准确反映

---

### 透明度与可访问性（ADR-990.5）

**规则**：

路线图 **必须**对所有团队成员可见，并易于访问。

**访问方式**：
1. **主路线图**：`docs/ROADMAP.md`（仓库根路径）
2. **README 链接**：项目 README 必须链接到路线图
3. **Wiki/文档站**：如有，同步更新
4. **团队会议**：季度评审会议讨论

**变更通知**：
- 路线图重大变更时，创建 Issue 通知
- 标签：`roadmap-update`
- 订阅者：所有团队成员

**反馈机制**：
```markdown
## 路线图反馈

对路线图有建议？请：
1. 创建 Issue，标签 `roadmap-feedback`
2. 描述建议内容
3. 说明优先级和原因
4. 等待季度评审讨论
```

**讨论平台**：
- **RFC**：重大变更提交 RFC
- **Issue**：小调整创建 Issue
- **会议**：季度规划会议集中讨论

**核心原则**：
> 人人可见，人人可参与。

**判定**：
- ❌ 路线图仅管理层可见
- ❌ 无反馈渠道
- ✅ 透明公开，欢迎反馈

## 破例与归还（Exception）

### 允许破例的前提

破例 **仅在以下情况允许**：
- 项目初期（<6 个月）路线图尚未稳定
- 团队规模过小（<3 人）无法支持季度规划
- 特殊情况（如公司重组）导致规划暂停

### 破例要求

每个破例 **必须**：
- 记录在路线图文档中，说明原因
- 标记预期恢复正常规划的时间
- 架构委员会批准

**未记录的破例 = 未授权治理违规。**

---

## 变更政策（Change Policy）

### 变更规则

本 ADR 属于 **治理层文档演进规则**：
- 修改需架构委员会同意
- 需通知所有团队成员
- 需更新相关流程文档

### 失效与替代

- 本 ADR 一旦被替代，**必须**更新路线图管理流程
- 不允许"隐性废弃"

---

## Non-Goals（明确不管什么）

本 ADR **不负责**：
- 产品功能路线图（仅管理文档体系演进）
- 项目管理细节（如日常任务分配）
- 团队绩效评估
- 预算和资源分配
- 技术栈演进路线图（可另行制定）
