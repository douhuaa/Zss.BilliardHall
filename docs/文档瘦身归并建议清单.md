## 🎯 单一真相源（SSOT）原则

### 什么是"单一真相源"？

> **定义**: 每个知识点、概念、流程只在一个权威文档中详细说明，其他地方只能引用，不能复制

### SSOT 实施规则

| 主题 | 唯一权威文档 | 禁止出现的地方 |

## 🔍 轻量审查机制

### 目标

- **防止文档膨胀**: 新增文档需要审查
- **维护 SSOT**: 修改文档时检查是否影响其他文档
- **低成本**: 不增加团队负担

### 审查流程

#### 1. 新增文档审查（PR 级别）

**触发**: 任何 `docs/` 目录下的新增 `.md` 文件

**检查清单**:
```markdown
## 📝 新增文档审查清单

- [ ] **必要性**: 此文档是否真的需要？能否归并到现有文档？
- [ ] **唯一性**: 是否与现有文档重复？（搜索关键词验证）
- [ ] **大小**: 预计行数是否 < 500 行？（超过考虑拆分）
- [ ] **位置**: 放在哪个目录下最合适？
- [ ] **链接**: 是否已在 README 或父目录索引中添加？
- [ ] **SSOT**: 是否声明了此文档的"主题边界"？
```

**决策**:
- ✅ 全部通过 → 批准合并
- ⚠️ 1-2 项未通过 → 要求修改
- ❌ 3+ 项未通过 → 拒绝，建议归并

#### 2. 文档修改审查（自动化）

**自动检查** (GitHub Actions):
```yaml
# .github/workflows/docs-check.yml
name: 文档审查

on:
  pull_request:
    paths:
      - 'docs/**/*.md'

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - name: 检查文件数量
        run: |
          file_count=$(find docs -name "*.md" | wc -l)
          if [ $file_count -gt 60 ]; then
            echo "❌ 文档数量超过目标 ($file_count > 60)"
            exit 1
          fi
      
      - name: 检查单文件大小
        run: |
          large_files=$(find docs -name "*.md" -exec wc -l {} + | awk '$1 > 1000 {print $2}')
          if [ -n "$large_files" ]; then
            echo "⚠️ 以下文件超过 1000 行，建议拆分："
            echo "$large_files"
          fi
      
      - name: 检查 Legacy 文件
        run: |
          legacy_files=$(find docs -path "*/legacy/*" -name "*.md")
          if [ -n "$legacy_files" ]; then
            echo "❌ 检测到 Legacy 文件，应删除："
            echo "$legacy_files"
            exit 1
          fi
```

#### 3. 季度文档健康检查

**频率**: 每季度一次

**检查项**:
- [ ] 文档总数是否超过目标 (60 个)？
- [ ] 是否有超过 6 个月未更新的文档？（考虑删除）
- [ ] 是否有新的重复内容出现？（搜索关键段落）
- [ ] README 链接是否全部有效？（自动化）
- [ ] 是否有新的 Legacy 内容产生？

**工具**: 使用脚本自动生成报告
```bash
# scripts/docs-health-check.sh
#!/bin/bash
echo "📊 文档健康检查报告"
echo "===================="
echo ""
echo "文件总数: $(find docs -name '*.md' | wc -l)"
echo "超过 1000 行的文件:"
find docs -name "*.md" -exec wc -l {} + | awk '$1 > 1000 {print $1, $2}'
echo ""
echo "6 个月未更新的文件:"
find docs -name "*.md" -mtime +180 -ls
```

## ⚠️ 风险与缓解

### 风险 1: 信息丢失

**风险**: 归并/删除文档时丢失重要信息

**缓解措施**:
- ✅ 在删除前创建 Git tag: `docs-backup-before-slim`
- ✅ 归并时逐段检查，不丢弃任何段落
- ✅ 保留 Legacy 到独立分支 `archive/docs-legacy`
- ✅ Code Review: 至少 2 人审查归并后的内容

### 风险 2: 链接失效

**风险**: 文档路径变更导致内部链接失效

**缓解措施**:
- ✅ 使用相对路径，不用绝对路径
- ✅ 归并后运行链接检查脚本
- ✅ 在旧文档位置添加跳转提示（保留 1 个月）
```markdown
# 本文档已归并

⚠️ 本文档内容已归并到 [新位置](../path/to/new-doc.md)

本文件将在 2026-02-13 后删除。
```

### 风险 3: 团队抵触

**风险**: 团队成员习惯旧结构，抵触变更

**缓解措施**:
- ✅ 提前沟通变更原因和好处
- ✅ 提供"新旧结构对照表"
- ✅ 分阶段执行，而非一次性大改
- ✅ 保留最常用文档路径不变（如快速上手指南）

### 风险 4: 大文件难以维护

**风险**: 归并后单个文件过大（如架构蓝图 3500 行）

**缓解措施**:
- ✅ 使用清晰的章节和锚点导航
- ✅ 在文档开头提供"跳转到章节"目录
- ✅ 可选: 保持多文件，但在 README 明确"主次关系"
- ✅ 定期评估: 如果 > 2000 行，考虑拆分
