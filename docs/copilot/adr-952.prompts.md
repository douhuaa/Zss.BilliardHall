## 概述

本文件为 GitHub Copilot 提供关于工程标准与 ADR 边界划分的场景化指导。

**核心原则（来自 ADR-952）**：
- ADR 定义"是什么"和"必须/禁止"
- 工程标准细化"实施细节"
- 工程标准不得引入新架构约束
- 工程标准基于 ADR 但不替代 ADR

---

## 场景 1：判断应写 ADR 还是工程标准

### 用户问："这应该写成 ADR 还是工程标准？"

**Copilot 应该**：

```
判断标准：

✅ 应该写 ADR：
- 定义架构约束（如"模块必须隔离"）
- 影响系统全局的决策
- 需要架构测试验证的规则

✅ 应该写工程标准：
- 细化 ADR 的实施细节（如"模块目录结构"）
- 代码风格和命名规范
- 工具配置和最佳实践
- 不引入新约束的细化说明
```

**示例**：
```
ADR：模块必须隔离（约束）
工程标准：模块目录结构应为 UseCases/Domain/Contracts（细节）
```
