## 🏗️ 核心原则

### 1. 有明确的信息架构

- ✅ 每个文档有清晰的定位和边界
- ✅ 13 个目录保持不变，但每个目录更精简
- ✅ 明确 3 个必读文档（README + 快速上手 + 架构蓝图）

### 2. 严格的"单一真相源"（SSOT）

> **定义**: 每个知识点只在一个权威文档中详细说明

**示例**:
```
Wolverine 架构 → 唯一权威: Wolverine模块化架构蓝图.md
代码风格     → 唯一权威: 06_开发规范/代码风格.md
Saga 使用    → 唯一权威: 06_开发规范/Saga使用指南.md
```

**引用规范**:
- ✅ 正确: `详见 [Saga 使用指南](路径)`
- ❌ 错误: 在多个地方复制完整说明

### 3. 轻量审查机制

#### PR 级别审查
新增文档时检查 5 项：
- [ ] 必要性：能否归并到现有文档？
- [ ] 唯一性：是否重复？
- [ ] 大小：< 500 行？
- [ ] 位置：放哪个目录？
- [ ] SSOT：是否声明主题边界？

#### 自动化检查
- 文件数 > 60 → 警告
- 单文件 > 1000 行 → 建议拆分
- 检测到 Legacy 文件 → 失败

#### 季度健康检查
- 文档总数趋势
- 6 个月未更新的文件
- 重复内容检测
- 链接有效性验证

## ⚠️ 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|-------|------|---------|
| 信息丢失 | 中 | 高 | Git tag 备份、逐段检查、独立存档分支 |

**总体风险等级**: 🟡 **可控**

**关键缓解措施**:
- ✅ 执行前创建 Git tag: `docs-backup-before-slim`
- ✅ 保留 Legacy 到独立分支: `archive/docs-legacy`
- ✅ 分阶段执行，每阶段后评审
- ✅ 至少 2 人 Code Review
